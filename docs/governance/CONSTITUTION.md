# 宪法原则

> 继承自原加气站项目 CONSTITUTION.md 的 8 条协作原则，并映射到新的 Agent-Driven 框架中。
> 所有 Agent、Skill、团队成员都必须遵守这些原则。

---

## 8 条原则及其在新框架中的体现

### 原则 1: 先分解再动手

- 禁止未理解全貌就直接编码
- 将大问题拆解为可管理的子问题
- 每个子问题须有明确的输入、输出、验收标准

**框架映射**: `requirement-decomposition` Skill 强制 L1-L4 分解，每个 L4（原子功能点）必须可被单个测试用例验证。

### 原则 2: 先标准后编码

- 大部分时间花在：建立标准、对齐方案、确保需求一致性、拆解需求、设计
- 用户深度参与标准和设计阶段；实现阶段参与度可降低
- 设计确认后方可编码

**框架映射**: `architecture.md` 为**阻断性门禁**（见 [AGENT-WORKFLOW.md](AGENT-WORKFLOW.md)），未经用户确认不可进入实现阶段。

### 原则 3: 忠实准确

- 分析、设计、实现必须忠实于已确认的需求和规格
- 禁止即兴发挥、假设、跳过
- 数据和结论必须可追溯

**框架映射**: Agent 生成的代码必须可追溯到已确认的需求（通过 RequirementTag + userStoryMapping）。

### 原则 4: 框架优先、MVP 优先

- 先搭整体产品框架和 MVP 原型
- 再优化细节和差异化功能
- 避免过早优化和过度设计

**框架映射**: 重构采用渐进策略——先试点站点管理模块 → 验证 → 推广到其余 5 个模块。

### 原则 5: 不确定须确认

- 任何未定义的概念或不清晰的问题必须与用户确认
- 禁止未经确认的假设
- 宁可多问一次，不可多错一步

**框架映射**: Agent 遇到未在 STANDARDS.md 中注册的术语或未在需求中定义的行为时，必须暂停并向用户确认。

### 原则 6: 纠偏经验持续积累

- 每次用户纠正都记录在独立文件中
- 简明版: `CORRECTIONS.md` — 模式查询表 + 摘要
- 完整档案: `CORRECTIONS-ARCHIVE.md` — 含上下文的完整记录
- 模式格式: `§ Section → P1-P10 模式编码 → 具体规则`

**框架映射**: 继承 CORRECTIONS.md 机制（见 [CORRECTIONS.md](CORRECTIONS.md)），每个新模块开发前必须复读模式表。

### 原则 7: 删除需双重确认

- **绝对禁止**未经确认就删除文件
- 至少需要**两次独立的用户确认**:
  1. 第一次: 明确告知用户要删除的文件/文件夹
  2. 第二次: 执行前提醒"此操作不可逆"
- 优先采用非破坏性方案

**框架映射**: 通过 SDK 的 `canUseTool` 回调，对文件删除操作实施双重确认拦截。

### 原则 8: 关键文档变更需用户审批后提交

- 需要审批的文档类型:
  - **规划类**: PLAN.md, ROADMAP.md, PROGRESS.md
  - **架构类**: architecture.md（任何模块）
  - **需求类**: requirements.md, user-stories.md
  - **技能类**: docs/skills/ 定义
  - **协作协议类**: CONSTITUTION.md, STANDARDS.md, AGENT-PLAN.md 等
- **流程**: 完成变更 → 展示给用户 → 用户审批 → Git commit
- 纯前端代码变更（组件、样式、i18n）可直接提交

**框架映射**: 通过 SDK 的 `hooks` 机制拦截关键文档的写入操作，要求用户审批后方可 commit。
