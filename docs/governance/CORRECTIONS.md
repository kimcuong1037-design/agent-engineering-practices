# 纠偏模式表

> **文档类型**: 活文档（随项目演进持续更新）
> **使用规则**: 每个新模块开发前，必须复读 §1 模式表作为前置检查。
> **继承自**: 原加气站项目 CORRECTIONS.md，初始化 10 个模式。

---

## §1 模式查询表

| 编码 | 主题 | 核心要点 |
|------|------|----------|
| **P1** | 路由/导航/命名一致性 | 1) 路由定义为常量（router.tsx）；2) 所有入口同步；3) 静态路由先于动态；4) 使用完整路由路径引用页面；5) 统一页面在文档中的命名 |
| **P2** | 模块交付检查清单（最后一公里） | 9 项检查：路由注册、导航菜单、面包屑、i18n、RequirementTag、Table scroll.x、architecture.md、跨模块视觉一致性、构建通过 |
| **P3** | 流程遵守与步骤 | 1) 严格顺序；2) 架构阶段完成实体定义；3) 设计阶段端到端走查；4) 修复须传播到方法论层面；5) 重大修复后全量影响扫描 |
| **P4** | 交互完整性 | 1) hoverable/cursor:pointer 必须有 onClick；2) 面包屑可点击；3) 统计卡片与列表过滤器双向联动；4) 主列表含全量数据，专项视图是过滤而非分区 |
| **P5** | 数据模型与 Mock 完整性 | 1) 核心实体架构阶段完整定义；2) Mock 覆盖全生命周期；3) 交叉验证状态与详情数据；4) 扩展 custom_fields/source_doc/tags；5) 明确"生成"含义 |
| **P6** | Skill 定义质量 | 1) 分解 Skill 须定义粒度标准；2) 明确 Agent 行为边界；3) 明确上下游依赖；4) 评估 Skill 须提供可执行验证步骤（grep 命令而非"检查 XX"）；5) 考虑是否需要 knowledge-base 输入 |
| **P7** | 跨阶段/跨模块依赖管理 | 1) 架构完成后执行跨阶段依赖审查；2) 4 层依赖标记 `⚠️ Phase N Dependency`；3) 最小侵入策略；4) 跨模块依赖须有 UI 入口；5) 上游 architecture.md 须列出下游消费者；6) 定义跨模块数据流触发机制；7) 反向影响审查 |
| **P8** | 修复优先级与决策框架 | 1) P1 分类：业务逻辑 vs 样式/一致性；2) 评估修改成本（低/高）；3) 执行前须用户确认；4) 页面合并判定：列表重叠 >60% + 操作重叠 >50% → 合并 |
| **P9** | i18n 翻译键命名规范 | 1) 禁止嵌套对象键作叶子翻译键；2) 使用 common.* 通用词；3) 叶子节点必须是字符串；4) 自测 "returned an object" 警告；5) 全局完整性校验；6) 禁止仅依赖 inline fallback |
| **P10** | API 文档——写操作需 JSON 示例 | 1) 所有写端点须有完整请求体示例；2) 须有响应示例（状态码 + 关键字段）；3) 无状态端点也须文档化响应格式；4) 字段约束紧邻示例；5) GET/POST/PUT/PATCH/DELETE 同等对待 |

## 纠偏记录规则

### 何时记录
- 用户指出 Agent 输出中的错误或不规范时
- 发现可归类为通用模式的问题时

### 记录格式
1. **§1 模式表**: 新增一行（编码递增 P11, P12...）
2. **§2 时间线**: 添加一行记录
3. **CORRECTIONS-ARCHIVE.md**: 写入完整上下文和根因分析（如需要）

### 模式升级
- 如果某个纠偏点可以归入已有模式 → 更新该模式的"核心要点"
- 如果无法归入 → 创建新模式编码
- 检查是否需要更新相关 Skill 定义（P6 模式要求）
